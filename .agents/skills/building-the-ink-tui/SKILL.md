# Building the Ink TUI

## When to use
Adding or editing the guided menu under `src/ui/`, a new screen/flow, the live progress panel, the invocation gate, or the bilingual strings — while keeping every flag/command working directly.

## Procedure / injected knowledge
- **No build step: htm, not JSX.** Author components as `` html`<${Box}>...</${Box}>` `` with `html = htm.bind(createElement)` (`src/ui/html.js`). Stack is `ink@^5` + `react@^18` + `@inkjs/ui@^2` + `htm@^3` (ink@7 needs react@19 and does NOT match @inkjs/ui@2). `import { createElement } from 'react'` works under Node ESM (cjs-module-lexer). htm has NO naked JSX fragment: `<>…</>` parses as tag `''` (React throws `InvalidCharacterError` → blank render) — use `<${Fragment}>…<//>`. `src/web-ui/app.js:5-8@47bfa19`. Verified: browser pageerror reproduced, then fixed.
- **Cores/glifos SEMPRE via tokens semânticos (`src/ui/theme.js`), nunca literais inline.** `colors` (accent=cyan valores/foco, title=magenta títulos/réguas, ok/warn/err, link, muted), `glyphs` (pointer ❯, app ◆, tick ✓ ≠ saved ✔ — não fundir, ambos assertados), `space`, `SPINNER_FRAMES` e `uiTheme = extendTheme(defaultTheme, {components: {...}})` recolorindo os widgets do `@inkjs/ui` p/ os tokens: `Select`/`MultiSelect` `styles.focusIndicator`/`label` (default é foco AZUL, briga com o accent), `Spinner` `styles.frame` (default azul), `ProgressBar` `styles.completed` (default MAGENTA). O extendTheme faz deepmerge — chaves default não sobrescritas sobrevivem; funções de estilo são FOLHAS (substituídas inteiras). `<${ThemeProvider} theme=${uiTheme}>` na raiz do App; componentes montados DIRETO em teste caem no defaultTheme (context default) → **nunca assertar COR em teste, só texto/glifo**. Widgets compartilhados em `src/ui/widgets.js`: `Panel` (borda round; política: só DUAS superfícies com borda no app — overlay `v` e o card do comando no Review; borda é significado, não decoração), `FooterHints` (pares `{k,label}` → `"k label · k label"`; `inline` devolve só o `<Text>` p/ compor na mesma linha; verbos são fragmentos i18n `hint_*` + `keySpace` nos DOIS blocos — matou as 5 strings de hint divergentes), `Header` (breadcrumb `◆ <título>` por mapa de tela; fora do mapa não pinta). `useSpinnerFrame` compartilhado vive em `src/ui/hooks.js`. `src/ui/theme.js@bdab7e5`, `src/ui/widgets.js@bdab7e5`. Verified: npm test 179 pass (`ui.theme` asserta os overrides contra os defaults do pacote instalado) + render TTY real (pt/en/NO_COLOR).
- **Checkbox multi-select = `@inkjs/ui` `MultiSelect` + `useInput` paralelo SÓ de Esc.** Padrão do `SourcesStep` (screens.js): `options` + `defaultValue` (todas marcadas → Enter direto = "todas", que o onSubmit converte em `null` = sem flag; subconjunto vira `--sources "A,B"` via join de vírgula). MultiSelect é UNCONTROLLED (`defaultValue` lido SÓ no mount) → o erro de validação ("0 marcadas") vive DENTRO do componente sem trocar `step` — remontar perderia a seleção; `onChange` limpa o erro ao re-marcar. Um `useInput` paralelo ouvindo APENAS `key.escape` é seguro com MultiSelect montado (o Ink transmite teclas a TODOS os hooks ativos; MultiSelect não tem campo de texto, mas espaço/letras no hook paralelo COLIDIRIAM com o toggle). Guard de lista vazia ANTES do MultiSelect (`options=[]` + Enter submeteria `[]` e prenderia o usuário no erro). `src/ui/screens.js@bdab7e5`. Verified: `test/ui.crawl-sources.test.js` (pré-marcadas, toggle+subset, todas→null, 0→erro sem submit, Esc) + coleta real via `--sources` (2 fontes exatas seedadas, US$ 0,14).
- **Testes navegam o menu por LABEL, não por DOWN×N.** `test/helpers/ink.js` (sem sufixo `.test.` → o runner ignora): `selectMenuItem(stdin, lastFrame, label)` desce até o ponteiro `❯` estar na linha do label (match de PALAVRA inteira — `Buscar` NÃO casa `Buscador web`) e dá Enter. Reordenar o menu ou inserir um passo não quebra mais `ui.search`/`ui.web`; `ui.crawl-input` segue keystroke-neutro porque Enter no MultiSelect todo marcado avança igual ao Select antigo. `test/helpers/ink.js@bdab7e5`. Verified: suites migradas verdes.
- **The no-build pattern has a BROWSER twin (`ncrawl web`).** `src/web-ui/` renders with React+ReactDOM UMD as globals + `htm.module.js` as an ES module, all served FROM node_modules by `src/web.js` (offline, no CDN). Exports maps BLOCK deep paths (react exposes no `./umd/*`; htm not even `./package.json`), so resolve via `path.dirname(require.resolve('<pkg>/package.json'))` — and for htm via `path.dirname(require.resolve('htm'))`. `src/web.js:17-25@47bfa19`.
- **Long-running command = its own screen.** RunView's contract is a FINITE thunk (resolve → Alert → back). The web server screen (`WebConfig` → `WebRun`) owns the lifecycle instead: `startWebServer` on mount, `close()` in the effect cleanup, `useInput` q/Esc→onBack and o→openBrowser. `src/ui/screens.js@47bfa19`. Verified: npm test navigates menu → port prompt (`test/ui.web.test.js`).
- **Commands live in `src/commands.js`** (side-effect-free), shared by the CLI (`src/index.js`) and the UI. The UI builds a thunk `() => cmdX(flags, rest)`; the command logs/persists exactly as on the CLI. `getStatus()` returns the counts as DATA (reused by the StatusBar and the run panel).
- **Lista navegável + preview NA MESMA tela = um ÚNICO `useInput` ramificando por modo.** A `ResultsView` (busca) virou o padrão: lista plana selecionável (buckets concatenados com anotação `_sec`; cabeçalhos são blocos NÃO selecionáveis — `blockOf(i)` compensa os offsets), seleção com auto-scroll que "puxa" o cabeçalho junto do 1º item da seção, Enter abre a PREVIEW como **estado interno** (`preview != null`), NUNCA um segundo componente com outro `useInput` (os dois receberiam as teclas). Preview = cabeçalho fixo + corpo `Box height=${H}` com `lines.slice(off, off+H)` (mesmo viewport do RunView), corpo quebrado por `wrapPlainText(text,width)` (exportada; cap 5000 linhas). Dimensões de `useStdout()` com clamps E fallbacks (`columns||80`, `rows||24` — ink-testing-library não tem rows). Efeitos colaterais entram por PROPS (`onOpen=openBrowser`, `getArticle` de commands.js) → o componente fica puro e o teste injeta spies sem DB/xdg-open. Updaters de estado FUNCIONAIS (rajada de teclas no mesmo tick usa closure velha). Teclas de letra (b/q/o/j/k) são seguras aqui porque NÃO há TextInput montado. `src/ui/ResultsView.js@6b1d77d`, `src/ui/App.js@6b1d77d`. Verified: `test/ui.results.test.js` (seleção, preview, `o` via spy, Esc, id sumido → fallback).
- **Live progress WITHOUT touching the crawl:** `setLogSink(fn)` (util.js) routes `log`/`warn`/`errorLog`/`debug` to the UI; `RunView` captures them in a BOUNDED ring buffer (~200 lines, NOT `<Static>` — it grows unbounded and lingers across runs) and polls `getStatus()` + `getRunTelemetry()` every ~300ms. `getRunTelemetry().progress` (o rastreador em memória `src/progress.js@3351a58`) alimenta as linhas AO VIVO do crawl: `agora:` (fases ativas — fetch/render/limpeza/curadoria/verificação/resumo/classificação), fontes x/y no counters, salvos/blurb/estouros, e o % por DATA rumo ao `--since` (global + as fontes mais atrasadas; ✓ = piso atingido; fontes sem data ficam fora da média). Essas linhas são GATED por `spec.sub === 'crawl'` — o snapshot de progresso é global do processo e um comando seguinte (classify/summarize) na MESMA sessão TUI herdaria progresso velho do crawl anterior. The command is I/O-bound, so the event loop stays free for Ink to render. Unset the sink in BOTH the promise `.finally` and the effect cleanup. A ResultsView/preview NÃO mexe no sink (nenhum comando roda ali). `src/ui/RunView.js@3351a58`. Verified: npm test (150 pass) + a linha `progresso:` equivalente do CLI ao vivo no run #3.
- **StatusScreen se AUTO-ATUALIZA** (interval de 1s re-lendo `getStatus()` — só COUNTs, barato): era um snapshot morto e um crawl rodando em outro processo nunca aparecia. Padrão p/ qualquer tela de leitura: receber o status inicial como prop e re-poll no próprio componente com cleanup do interval. `src/ui/screens.js@3351a58`.
- **Invocation gate (`src/index.js`):** open the UI on `ui`/`menu` OR (no args AND `stdin.isTTY && stdout.isTTY` AND not `--no-input`/`NO_INPUT`); otherwise dispatch the flags directly; bare + non-TTY -> `printHelp()` (NOT crawl). Load Ink via dynamic `import('./ui/index.js')` so the CLI path never pays the React cost.
- **Lifecycle (`src/ui/index.js`):** `render(html\`<${App}/>\`, {patchConsole:true})`, `await waitUntilExit()`, then `setLogSink(null)` + `closeBrowser()` + `db.close()` + `exit(0)`. `exitOnCtrlC` (default true) restores the terminal on Ctrl-C; an abandoned `in_progress` job resumes next run via `resetInProgress`.
- **Focus + input identity:** @inkjs/ui `Select`/`TextInput` capture input while mounted, so render ONE input at a time (wizard by `step`). But `TextInput` is UNCONTROLLED — its buffer is seeded from `defaultValue` only at mount and never re-synced — so consecutive `step`s that render `<Field><TextInput/></Field>` at the SAME tree position with the SAME type make React reuse the fiber and the 1st field's text BLEEDS into the next (and a bled non-numeric value then trips `parseIntFlag`). Fix: give each `TextInput` a `key=${step}` (a different key forces unmount+remount = fresh empty buffer); `key` on the wrapper `Field` also works. Keep `step` stable within a step (a `setErr` must NOT change `step`) so mid-typing validation preserves the in-progress text. Do NOT register global plain-letter hotkeys (they steal `TextInput` chars). Pre-validate in the screens so the commands' `process.exit` guards (bad `--since`, missing url, `reset --yes`, no LLM) are unreachable under Ink.
- **i18n + transparency:** `t(key)` is PT default / EN via `CRAWLER_LANG` (UI chrome only; crawl logs stay PT). Novas chaves SEMPRE nos DOIS blocos (pt e en). Always show the equivalent command (`commandPreview`) on the review screen so users learn the flags.
- **O painel do CRAWL é um DASHBOARD, não um fluxo de log.** `RunView` virou um DISPATCHER fino (thunk/poll/sink/subscribe/teardown): p/ `spec.sub==='crawl'` renderiza `CrawlDashboard` (região de STATUS persistente — cabeçalho+badge de estado+cronômetro, TABELA DE FASES com `@inkjs/ui` `ProgressBar`, linha "agora", % por data, faixa de métricas — SEPARADA de um FEED curado + rodapé); os demais comandos caem no `GenericRun` (o painel simples de antes, com `searchScanning`+`onResults`; a BUSCA agora ganha `ProgressBar` (@inkjs/ui, `Box` com `width`) + % + ETA (via `startAt.current`) + "N não analisados" (`prog.failed`) SÓ quando `prog.total>0` — progresso nível-ARTIGO; o modo B não move `_progress` → total=0 → fica indeterminado, só spinner. `src/ui/RunView.js@40670d4`, `i18n.js@40670d4` (`searchEta`/`searchFailed`, PT+EN)). Padrão Turborepo/BuildKit: separar o STATUS agregado do LOG detalhado é o que mata a confusão de "quando começa/termina cada processo". A derivação das fases é PURA (`src/ui/crawlPhases.js` — `derivePhases`/`deriveBadge`, sem Ink; `result` do RunView decide done/failed, NÃO a presença de erro no feed — falha de job isolado não reprova a run). `src/ui/CrawlDashboard.js`, `src/ui/RunView.js`, `src/ui/runLines.js@<this>`. Verified: `npm test` (`ui.crawl-dashboard`, `ui.runview`, `crawlPhases`).
- **Marcos ao vivo = camada de EVENTOS estruturada, separada dos contadores (`progress.js`) e do log cru (`setLogSink`).** `src/run-events.js` (módulo PURO, sem imports → sem ciclo; `progress.js` pode importá-lo e emitir `source-done`/`floor-hit` na transição): `emitRunEvent({phase,kind,source?,detail?,level?,channel?})` + `subscribeRunEvents` + `runEventsReset` (chamado no `crawlRun`, ao lado de `progressReset` — o ring é global ao processo, não pode vazar entre runs). Ring cap 100 + coalescing de rajada idêntica (`×N`); canal `ticker` = "último salvo" (atualiza no LUGAR, não polui o feed). MARCOS (fase início/fim, fonte concluída, coletânea curada, timeout, erro) vão pro feed; RUÍDO interno (`warn`/plumbing: parse-pool/governor/breaker) COLAPSA num contador `⚠ N avisos internos` com toggle `v` (overlay do log cru). Escolhido a classificar strings PT do `log()` — marcos e plumbing são AMBOS `level:'log'`, então separar por regex de PT seria frágil. Verified: `test/run-events.test.js`.
- **Dois gotchas de Ink VERIFICADOS:** (1) **todo timer de ANIMAÇÃO precisa de `.unref()`** — um `setInterval` de spinner NÃO-unref'd segura o processo vivo e PENDURA o `node --test` (isolation=process) depois que os testes terminam (o poll de dados de 300ms idem; o Ink já mantém o loop enquanto renderiza). (2) **`@inkjs/ui` `Badge` deixa o texto em MAIÚSCULAS** (`Concluído ✓` → `CONCLUÍDO ✓`) — asserte case-insensitive nos testes. Corolários: prefira UM frame de spinner COMPARTILHADO (não N `<Spinner>`, cada um com seu timer → dessincronia/flicker; o Ink redesenha a árvore viva inteira a cada state change); glifos (`✓ ✗ · ▶ ⏱ ⚠ ▤`) carregam o estado como canal redundante p/ `NO_COLOR`; `ProgressBar` só recebe `value` (0–100) e precisa de um `Box` com `width` (senão não aparece).
