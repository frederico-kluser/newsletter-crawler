# Building the webapp frontend

## When to use
Editing anything under `webapp/src/` — a UI string/translation, the language toggle, the onboarding tutorial, a theme token, a Motion animation, or a React component. NOT the AI-search rubric/port or the export snapshot contract (those stay in searching-the-corpus).

## Procedure / injected knowledge
- **Separate app, and it uses REAL JSX (unlike the TUI).** `webapp/` is its own Vite + React 19 + Motion project (`webapp/package.json`), authored in normal `.jsx` that Vite compiles. Do NOT bring the TUI's `htm` tagged-template no-build pattern here — that is `src/ui/`, a *different* app (see building-the-ink-tui). It is a 100% STATIC site: reads a committed JSON snapshot (`webapp/public/data/{meta,articles,contents}.json`), no backend, no env vars. `MotionConfig reducedMotion="user"` at the root (`webapp/src/main.jsx`) makes every Motion animation respect the OS setting; animate ONLY `transform`/`opacity`/`filter`.
- **Semantic design tokens are the ONLY thing a component reads for color.** `webapp/src/styles/tokens.css` has 3 layers: raw primitives → SEMANTIC (`--bg`,`--fg`,`--accent`,`--hairline`,`--sheet-bg`,`--seg-*`,`--overlay`…) → component. Dark overrides ONLY the semantic layer, via `:root[data-theme='dark']` (the toggle writes it) OR `@media (prefers-color-scheme: dark)` when the user never chose — mirroring the pre-paint script in `webapp/index.html`. Build any new surface with semantic tokens alone and it works in light AND dark for free (verified: the tutorial + language toggle theme correctly in both with zero dark-specific CSS). One spring vocabulary: `webapp/src/motion/transitions.js` (`springs.snappy/gentle/sheet`, `fades`).
- **i18n is a CUSTOM React Context — deliberately NOT react-i18next** (tiny app, strings already centralized; a heavy dep would fight the repo's zero-unnecessary-deps grain). `webapp/src/strings.js` exports `DICTS = { pt, en }` — every UI string plus the `KIND_LABEL`/`VERIFY_LABEL`/`FACET_LABEL` maps, per locale. `webapp/src/i18n.jsx` = `LocaleProvider` + `useStrings()` (the active dict) + `useLocale()` (`{locale,setLocale,toggle,strings}`). **Low-churn conversion pattern:** swap `import { STR } from '../strings.js'` for `const STR = useStrings();` in the component body — every `STR.x` reference stays byte-identical; pull label maps via `const { KIND_LABEL } = STR;`. Switching locale re-renders consumers (NO remount) so search/filter/scroll state survives the switch.
- **THE RULE: every UI string lives in BOTH dicts.** `webapp/test/i18n.test.js` fails if any key exists in `pt` but not `en` (or vice-versa), plus label-map keys and tutorial-step parity — so a missing translation is a RED TEST, not a silent Portuguese-in-English leak. Article CONTENT (original titles, `summary_pt`, tags) is DATA and is never translated — only the UI chrome is.
- **Browser-language detection = `resolveLocale()` (pure, testable, in `webapp/src/lib/locale.js`).** Product rule: the BCP-47 **primary subtag** (before the first `-`) equal to `pt` → `'pt'`; ANY other language → `'en'` (Portuguese, or the-rest-of-the-world-in-English). Use exact primary-subtag match, NOT `startsWith('pt')` — `'ptx'` is not Portuguese (that bug was caught by the parity/detection test). A saved choice (`localStorage nc-locale`) wins over detection. **This rule is DUPLICATED** in the inline pre-paint `<script>` in `webapp/index.html` (sets `<html lang>` before first paint, zero FOUC) — edit one, edit the other.
- **Locale-aware formatters via a module singleton.** `fmtUsd`/`fmtInt`/`fmtEta` (`strings.js`) and `fmtDate` (`lib/format.js`) render in the active locale (`1.588` vs `1,588`; `US$ 20,56` vs `US$ 20.56`; month abbreviations) by reading a module-level `_fmtLocale` set through `setFmtLocale`/`getFmtLocale` — because they are pure functions called OUTSIDE React (e.g. `AnimatedCount`'s default formatter is `fmtInt`, so counts localize too). The provider sets it in render so the first paint is already correct.
- **Onboarding tutorial = Apple "Welcome" sheet** (`webapp/src/components/Tutorial.jsx`). Centered card; steps come from `DICTS.*.tutorialSteps` (data-only `{icon,title,body}` — the SVG per `icon` id lives in the component); progress dots; Skip / Back / Continue / Get-started. **`AnimatePresence mode="wait"` is REQUIRED — do NOT switch to sync/popLayout:** under fast clicks the sync modes pile exiting step-nodes into the same grid cell (all 5 titles present at once) and pollute the focus trap (verified: Playwright strict-mode "resolved to 5 elements" failure). Accessible per W3C APG: `role=dialog`+`aria-modal`, focus trap on Tab, ←/→ navigate, Esc closes, focus returns to the trigger on unmount, `document.body` scroll locked. Auto-opens on first visit (`localStorage nc-tutorial-seen`); the topbar `HelpButton` (?) reopens it.
- **Two live `LanguageToggle` instances → `layoutId` MUST be a prop.** The tutorial overlay covers the topbar, so a second copy of `LanguageToggle` lives INSIDE the tutorial card (a wrong-language first-run user can switch during onboarding). Both mount at once, and the sliding "pill" uses Motion `layoutId`, which must be UNIQUE across the tree — so `LanguageToggle` takes `layoutId` (default `'lang-pill'`; the tutorial passes `'lang-pill-tut'`). Duplicate `layoutId`s make the shared-layout pill teleport between the two.
- **localStorage is fail-open.** `webapp/src/lib/storage.js` wraps every `get/set/del` in try/catch → falls back to an in-memory `Map` (Safari private mode THROWS on access). Keys: `nc-theme`, `nc-or-key`, `nc-locale`, `nc-tutorial-seen`. Persist new state through here, never raw `localStorage`.
- **The AI search + snapshot contract is a DIFFERENT skill.** `webapp/src/lib/{search,openrouter,filters,lane,pool}.js` are verbatim ports of the CLI rubric/schemas/`WEB_WHERE`/AIMD lane and are owned by searching-the-corpus (incl. the "edit both copies" sync gotcha and the `meta.search` export contract). Those AI PROMPTS are Portuguese LLM prompts, NOT UI chrome — they do NOT go through `DICTS`/i18n.

## Verification
`cd webapp && npm test` (pure libs + `test/i18n.test.js`) + `npm run build` (Vite compiles every `.jsx`). For UI behavior, drive the built app: `vite preview` + Playwright — a fresh context with `locale:'en-US'` vs `'pt-BR'` asserts detection, then click the in-card toggle, walk the steps, assert persistence (`nc-tutorial-seen`) + reopen via `HelpButton`; `colorScheme:'dark'` proves the token theming.
