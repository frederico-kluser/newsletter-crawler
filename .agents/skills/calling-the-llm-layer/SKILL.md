# Calling the LLM layer

## When to use
Editing `src/llm.js`, changing any OpenRouter/DeepSeek call, prompt, json schema, reasoning effort, or model; or debugging a 400/401 from OpenRouter.

## Injected knowledge
- **Reasoning effort: Pro uses `xhigh`, Flash uses `high` (mechanical stages can use `medium` — accepted by DeepSeek V4, probed HTTP 200). Never `max`** (HTTP 400; guard rebaixa p/ xhigh). Send only the nested `reasoning` object. `src/llm.js:28-29@79fd5d8`, `src/llm.js:139-146@d17d599`, `config/models.json:14-18@d17d599`.
- **Two-tier strategy.** Pro (xhigh) derives reusable selectors and classifies: `deriveLinkSelector`, `deriveContentSelector`, `classifyFacet`, `mapQueryToFacetTags`. Flash does the cheap high-volume work: `deriveNextLink`, `extractLinksItemByItem`, `extractArticleViaLLM`, `extractRoundupLinks`, `summarizeArticle`, `judgeRelevance`, **`judgeRelevanceBatch` (busca soft da web, xhigh)**, `curateRoundupItems`/`curateLeftoverLinks` (curadoria, high), `cleanArticleContent` (limpeza, medium), `verifyRecordLLM` (verificação, high), `deriveDateSelector` (spec de data CSS+regex, high) — stages resolved via `stageModel()` (config/models.json + env). `src/llm.js:359-600@d17d599`, `src/config.js:145-201@d17d599`.
- **Classify cost (multi-faceta = ALTO volume, cuidado).** `classifyOne` (`src/classify.js:17-47`) faz **1 chamada por faceta** (`getFacets()`, 9) por artigo — SEM lote. Modelo POR FACETA (`classifyFacetModel`, `src/config.js:292`): sem override herda a etapa base `classify`; `classify:<faceta>` (models.json) ou env `LLM_MODEL_CLASSIFY_<FACETA>`/`LLM_EFFORT_CLASSIFY_<FACETA>` trocam. Anti-padrão custoso (já foi ~92% do gasto de uma coleta longa): 6 facetas no **Pro/xhigh** com o **corpo inteiro reenviado 9×**. Perfil barato atual (config/models.json): só as CORE `domain`+`topic-technology` no Pro/high; as outras 7 em Flash/medium; input recortado por `CLASSIFY_MAX_CHARS`=2000. Pro≈10×Flash/chamada (seeds `src/budget.js:12`) — meça com `ncrawl finish --no-verify --no-summarize` (só o sweep de classify; byStage no extrato). Salto FUTURO (deferido): classificar em LOTE de artigos por chamada reusando `judgeRelevanceBatch`+`mergeBatchVerdicts`.
- **SMALL-OUTPUT RULE (hard-won).** A high-volume stage must return SMALL DATA (spans, items, verdicts), never long rewritten text: the first `articleClean` asked for the full cleaned body back and STARVED the pipeline — 18 in-flight calls, ZERO commits in 8+ minutes (reasoning + thousands of output tokens per call, then 180s timeout + retry chains) while a trivial probe answered in 2s. Redesigned to verbatim `junk_spans` (local deletion), calls land in seconds. Cost per article: ~$0.001. `src/llm.js:466-530@d17d599`, `src/clean.js:38-60@d17d599`. Verified: live A/B nos runs 3 (travado) e 4+ (fluido).
- **Batch judging (stage `searchBatch`): rubrica compartilhada + eco de id + fusão tolerante.** `judgeRelevanceBatch({query, items})` julga ~40 artigos por chamada (itens = 1 JSON por linha `{id,title,summary}`; saída pequena `{results:[{id,relation,kind}]}` — respeita a small-output rule). A rubrica/few-shots são a MESMA const `RELEVANCE_RUBRIC` do `judgeRelevance` unitário (texto escolhido por eval — F1 0.848 vs 0.731; NÃO reescrever sem re-rodar o eval), então soft e deep julgam com o mesmo critério. O prompt exige eco EXATO do id, mas a garantia real fica no CHAMADOR: `mergeBatchVerdicts` (search.js) trata id faltando→`none` fail-open, id inventado→ignorado, duplicado→1º vence; zod usa `z.coerce.number().int()` no id (o modelo às vezes ecoa string) e clamps de relation/kind. `src/llm.js@6b1d77d`, `src/search.js@6b1d77d`. Verified: `test/search.batch.test.js` + busca soft real (362 artigos/10 lotes, 0 ids perdidos).
- **Structured output + defensive parse + retry/escalation.** `response_format` json_schema `strict:true` + zod em TUDO; enums NÃO vão no json_schema (suporte não comprovado) — o clamp fica no zod (`kind` desconhecido -> 'news', `verdict` -> 'suspect'), com backstops determinísticos fora do LLM (SPONSOR_RE, isRealRecoveredItem, applyJunkSpans+sanityCheckCleaned). `callJSON` re-samples até `retries` no MESMO modelo e escala p/ Pro na última tentativa (Flash trunca JSON às vezes). `src/llm.js:114-172@d17d599`.
- **Roundup curation is PER-SECTION now.** `splitIntoSections` (curate.js) fatia a edição por seção (heading markdown / negrito / rótulo com emoji reconhecidos por `sectionTitleOf`) e dispara 1 `curateRoundupItems` por seção EM PARALELO, cada um com um `sectionHint` (Releases -> tende a release, Code&Tools -> tool, etc. — só um viés, o agente decide item a item). Mais paralelismo intra-edição (uma issue vira ~3–5 agentes) e prompts especializados; sem seções detectáveis, cai p/ chunk por tamanho. Guards do sectionTitleOf (verificados por repro): linha com URL/`](` NUNCA é rótulo (heading de ITEM como `## [Deno 2.9](url)` fatiava o item p/ fora do contexto) e rótulo solto não termina em `[.!?…]` (frase de prosa com palavra de seção virava seção). `src/curate.js@40d0505`, `src/llm.js@40d0505` (`sectionHint`/`curateRoundupItems({section})`). Verified: run #12 — 629→3, 630→5, 631→3 seções; 67 itens (vs 59 no modo chunk).
- **Curator recall is NOT guaranteed** — even with a good prompt the first pass omitted mid-page headline items; nunca confie numa única passada p/ completude. O par determinístico (diferença de conjuntos de links sobre o HTML BRUTO) + passe de cobertura dedicado (com a página INTEIRA podada como contexto) é quem garante. `src/curate.js@40d0505`. Verified: coverage event mostra leftovers/recovered por URL; os 3 itens que o Readability escondia entraram.
- **Transport.** SDK openai -> OpenRouter baseURL; `maxRetries:1` + `LLM_TIMEOUT_MS` 180s; penalidade de 429 COMPARTILHADA (backoff exponencial, teto 60s) + `reportRateLimit` halva a lane llm; `usage: {include: true}` traz o custo REAL p/ o ledger (commit ANTES do parse — 200 malformado também custou). `callJSON`/`createOnce` aceitam `signal` (AbortSignal do job, repassado como request option do SDK: `create(body, {signal})`): o teto duro do job cancela a chamada EM VOO e devolve o slot da lane; o retry de 429 e cada tentativa checam `signal.aborted` antes de admitir de novo. Entry points do job de artigo já plugados: `cleanArticleContent`/`deriveContentSelector`/`extractArticleViaLLM` (`{signal}` opcional — chamadas avulsas seguem sem). `src/llm.js:14-103@d17d599`, `src/llm.js@3351a58`. Verified: run #3 real (0 zumbis segurando a lane llm) + npm test verde.
- **Token discipline.** `pruneForLLM` + clamp (`MAX_HTML_FOR_LLM`, `CLEAN_MAX_CHARS`, `VERIFY_MAX_CHARS`, `SEARCH_MAX_CHARS`) antes de enviar. Curadoria usa MARKDOWN do corpo Readability (5KB limpos numa issue de 140KB de HTML). Lote da busca soft: título ≤200 + summary ≤400 chars por item. `src/curate.js:108-115@d17d599`.
- **The `.env` key override gotcha + runtime key.** `config.js` loads repo `.env` then `NC_HOME/.env` OVERRIDING shell vars — a stale exported key otherwise yields `401 "User not found"`. Probe: `curl -s https://openrouter.ai/api/v1/key -H "Authorization: Bearer <key>"`. `OPENROUTER_API_KEY`/`HAS_LLM` são **`export let` (live bindings ESM)** + `setRuntimeKey(k)`: a web UI (POST /api/key) e o `cmdKey set` ativam a key SEM reiniciar e sem tocar nenhum call site; o `client()` cacheia por key (`_clientKey`) e reconstrói quando ela muda. NÃO capture a key numa const local ao importar. `src/config.js@6b1d77d`, `src/llm.js@6b1d77d`. Verified: `test/config.key.test.js` + key/status dinâmico em `test/web.search.test.js`.
