# Persisting and orchestrating

## When to use
Editing `src/db.js` (schema/statements), `src/crawl.js`/`src/curate.js` (frontier/curation/pagination), `src/events.js` (event buffer), or `src/index.js` (loop); touching dedup or the queue.

## Injected knowledge
- **Schema.** Tables `sources`, `pages`, `articles`, `selectors`, `frontier`, `runs`, `llm_usage`, `classifications`, `article_tags`, `classification_uncovered`, **`events`** (run_id, source_id, url, stage, status, detail JSON — per-item pipeline trace read by `ncrawl inspect`); `UNIQUE(url)`; UNIQUE `content_hash`; WAL + foreign_keys + busy_timeout. `src/db.js@d17d599`.
- **Event writes are BATCHED (fail-open).** `logEvent` pushes to an in-memory buffer; the flush inserts the whole buffer in ONE better-sqlite3 transaction — on `EVENTS_FLUSH_AT` (50) or the explicit `flushEvents()` in `runWithLimits`'s `finally`. Cuts the fsync of thousands of tiny inserts. Trade-off: a HARD kill loses the un-flushed tail (telemetry only). A separate crawl process reading events (inspect) sees only committed rows. `src/events.js@40d0505`, `src/commands.js@40d0505`. Verified: `test/events.buffer.test.js` (auto-flush at threshold + explicit flush).
- **Articles quality columns (ensureColumn):** `kind` (news|tool|release), `issue_url`/`section`/`blurb`, `content_source` ('aggregator'|'target'), `needs_enrich`, `cleaned`, `verify_status`/`verify_notes`. Selectors gain `date_selector/date_attribute/date_regex`. All migrations idempotent via `ensureColumn`; `wipeAll` includes `events`. `src/db.js:137-186@d17d599`.
- **All SQL via one `stmts` object.** `claimNext` atomic UPDATE...RETURNING orders `retries ASC, id ASC`; `enqueue` INSERT OR IGNORE; ONE `insertArticle` serves both flows (needs_enrich distinguishes); `enrichArticle` fills target body WITHOUT touching kind/blurb/section; `requeueUrl` re-activates a done/failed job for a still-needs_enrich row; `requeueNeedsEnrichForSource` (run in the seed loop) re-activates ALL still-blurb-only article jobs of a source at crawl start (this is what makes the deadline's "enrich later" real). `src/db.js:68-132@79fd5d8`, `src/db.js:262-296,419-423@d17d599`, `src/db.js@40d0505`.
- **Busca: stmts de escopo + âncora do delta.** Os 4 stmts de busca (`listAllArticlesForSearch`, `listRunArticlesForSearch`, `articlesByTags(ForRun)`) trazem `source_name` (LEFT JOIN sources) + `date_iso` (`coalesce(iso_date(published_at), date(extracted_at))`) p/ a lista/preview da TUI. A busca IA da web tem WHERE próprio `SEARCH_SCOPE_WHERE` (`@sources` array JSON via json_each + from/to via iso_date) usado por `webSearchScopeCount`/`webSearchCandidates` (hard: content inteiro)/`webSearchCandidatesLite` (soft: summary_pt/blurb/substr(content,400)); hits re-selecionados por `webArticlesByIds` (mesmas colunas dos cards; ordem restaurada em JS via Map). **Âncora do escopo "apenas o novo" = `maxArticleRunId` (`MAX(articles.run_id)`), NUNCA `MAX(runs.id)`**: search/verify/web-search também abrem runs mas não setam `articles.run_id` — ancorar nelas zera o delta (bug real: MAX(runs.id)=3 vs artigos todos na run 1). `avgUsageByStage` (média real de custo por stage em llm_usage) alimenta as estimativas de US$ pré-busca. A função SQL `fold` foi REMOVIDA junto com a cláusula `@q` do `WEB_WHERE` (a busca por texto virou IA); `@kind` é CASE de 3 vias (release = coluna exata; release segue contando como tool). `src/db.js@6b1d77d`, `src/commands.js@6b1d77d` (`getSearchScope`). Verified: `test/web.search.test.js` + npm run status.
- **Register-at-curation, enrich-at-fetch, curated in ONE transaction (index sources).** `processRoundup` -> `curateRoundup`: the issue page becomes Readability markdown, split BY SECTION (`splitIntoSections`, 1 PARALLEL Flash agent per News/Tools/Releases/IN BRIEF) — see calling-the-llm-layer. Each curated item is INSERTED immediately (content = title+blurb, `needs_enrich=1`, published_at = issue date) and enqueued; the whole item loop runs inside a SINGLE `db.transaction` (batched writes; the loop is sync, logEvent only buffers). Enrichment keeps the blurb on thin/blocked/dup-content/timeout targets — a tool never disappears. Sponsors/jobs never saved (LLM label + deterministic SPONSOR_RE/JOB_RE). COVERAGE pass over RAW-page links (not the Readability body — it drops ad-adjacent blocks) + `isRealRecoveredItem` filter; the leftover agent gets the WHOLE pruned page as context. Curated items are exempt from the --since floor (issue is the anchor). `src/curate.js@40d0505`, `src/crawl.js:360-410@d17d599`. Verified: run #12 — 3 issues -> 3/5/3 sections, 67 items (kind+section), 0 sponsors, 0 docs-secondary; the 3 headline items Readability hid were all captured.
- **Frontier state machine + DEADLINE de TRABALHO com abort real.** `pending -> in_progress -> done | failed`; resume: `resetInProgress()` flips stale in_progress back to pending; `bumpRetry` até MAX_RETRIES; BUDGET_EXCEEDED devolve a pending sem consumir retry. O deadline do job de artigo NÃO é tempo de parede: `createJobClock(JOB_TIMEOUT_MS)` (`src/deadline.js@3351a58`) conta SÓ as fases de trabalho fetch/render/parse (`clock.run`); esperas de fila (lanes/politeness — `clock.wait`/`noteWait`) e as fases LLM ficam FORA do relógio (têm `LLM_TIMEOUT_MS`/orçamento próprios). Expirou => o clock ABORTA um AbortSignal e got/render/parse/LLM em voo PARAM, devolvendo as lanes — o `withTimeout` puro anterior não cancelava nada e o job estourado virava ZUMBI segurando as lanes, derrubando os seguintes em cascata (run real #1: 194/194 artigos estouraram, 0 salvos; com o clock, run #3: 0 estouros, 9 enriquecidos). `JOB_HARD_TIMEOUT_MS` (default 10x, 0=off) segue como teto DURO de parede via `withTimeout` por cima e, ao disparar, também aborta o clock (sem zumbi no backstop). O veredito de timeout no dispatch vem de `clock.expired()` (o abort pode aflorar como erro de cancelamento do got/SDK, não como code JOB_TIMEOUT). Timeout de enrich mantém frontier `done` + `needs_enrich=1` (re-enfileirado no próximo crawl); o evento `job/timeout` agora carrega `phases/waits` do clock — `ncrawl inspect` mostra ONDE o tempo foi. listing/roundup seguem ISENTOS do corte (cortar curadoria no meio joga fora o fan-out — bug real que já tivemos). `src/commands.js@3351a58`, `src/fetch.js@3351a58`. Verified: `test/deadline.clock.test.js` + `test/commands.timeout.test.js` + run #3 real (0 timeouts).
- **One `runs` row per command.** `runWithLimits`/`beginRun` opens the ledger run; crawlRun REUSES it as the delta watermark (`getBudgetState().runId`). A busca IA da web abre o run `web-search` por um mini-envelope PRÓPRIO no web.js (`withSearchRun`: initGovernor llm-only + beginRun/endRun; serializado — 1 busca por processo, 2ª leva 409) porque importar `runWithLimits` criaria ciclo commands.js↔web.js. `src/commands.js:127-140@d17d599`, `src/web.js@6b1d77d`.
- **Concurrency loop + STREAMING verify.** capacity = fetch+render lanes (governor); inflight Set + Promise.race. After a successful article save/enrich (processArticle returns `{verifyUrl}`), the loop schedules `verifyArticleRow` on the llm lane in a SEPARATE `verifying` set (does NOT count against fetch/render capacity); the loop waits on `[...inflight, ...verifying]`. So most fichas are verified DURING the crawl; the post-crawl `verifyPending` sweep stays as an idempotent (NULL-only) safety net for blurb-only leftovers. O crawl também alimenta `src/progress.js@3351a58` (em memória, sem SQL): fases ativas (fetch/render/limpeza/curadoria/verificação/resumo/classificação), contadores (salvos/enriquecidos/blurb/estouros/itensCurados), fontes x/y (`sourceSeen` no seed, `sourceListingDone` no dispatch) e o % por DATA rumo ao `--since` (`dateSeen` nos itens datados/published; `floorHit` nas paradas por piso => 100%) — lido por `getRunTelemetry().progress` (painel TUI) e pela linha `progresso:` periódica do CLI (junto do costTimer). `src/commands.js@40d0505`, `src/verify.js@40d0505`. Verified: run #12 — ~63 streaming + 4 sweep = 67 fichas verified, 60 ok/7 suspect; run #3 (linha progresso ao vivo) + `test/progress.test.js`.
- **Date floor / dedup / purge / pagination / inspect** — unchanged: `--since` pairs listing links with dates and stops pagination (curated items exempt); dedup = canonical URL + UNIQUE url + UNIQUE content_hash (curated content = `title — blurb`); `ncrawl purge <src> --yes [--selectors]`; pagination stops on empty/repeat-hash/no-next/`added===0`/below-floor/maxPages; `ncrawl inspect` (latest CRAWL run via `getLatestCrawlRunId`) prints the per-issue tree + per-stage cost + skip reasons, `--url` the event timeline. `src/crawl.js@d17d599`, `src/commands.js:427-560@d17d599`.
